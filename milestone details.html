PHASE 1 SCOPE

What We're Building First

Same goal you laid out, kept tight. Milestones 1 and 2 build the foundation and the canvas. The rest of the loop (AI routing, task execution, Answer Memory) sits in Milestones 3 to 5. Support Requests, the Operator Review Queue, Quick Send, export jobs, scheduled jobs, and semantic search all stay deferred until the core loop is proven, as we agreed.

1
The operator creates Function Blocks on the canvas.
2
They connect blocks with Process Lines.
3
They configure each line's Handler, Approver, and CC steps.
4
An end-user submits a messy request.
5
AI suggests a route from the configured Process Lines.
6
The receiving user responds, comments, approves, or requests changes.
7
The request owner sees the final answer.
8
The system creates a basic Provisional Answer Memory.

Two Things You Asked Me to Pin Down

CANVAS IN PHASE 1

Simple but real: create a block, drag it, connect blocks with a Process Line, save positions, click a line to configure it, and open the step editor for Handler / Approver / CC. No advanced visual features (no auto-layout, minimap, grouping, or animated edges) in Phase 1.

ANSWER MEMORY: BASIC RETRIEVAL

On case close, the resolved answer plus its request context is saved as Provisional Answer Memory. On a new request, the system surfaces similar past answers with a keyword and field match (Postgres full-text, scoped to the workspace and Process Line). No embeddings or semantic ranking yet; that comes later.
