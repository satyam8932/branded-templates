PHASE 1

What Phase 1 Proves

Phase 1 isn't a thin slice of the 60-page spec. It's the whole core loop working start to finish, built on a configurable foundation instead of hard-coded departments. All of this has to work before anything later (Support Requests, the Operator Review Queue, exports, and the rest) is worth building on top of it.

1
The operator builds the workflow. They create Function Blocks, drop them on the canvas, connect them with a Process Line, and set up its Handler, Approver, and CC steps.
2
An end-user submits a messy request into that workflow.
3
AI reads the request, works out what's actually being asked, and suggests a route from the Process Lines the operator set up. It doesn't lean on a hard-coded department list.
4
The receiving user responds. They comment, approve, reject, or ask for changes.
5
The request owner sees the final answer and closes the case.
6
The answer gets saved as Answer Memory, so a similar request later can reuse it instead of starting from scratch.

Deliberately Deferred

To keep Phase 1 from bloating, it leaves a few things out: Support Requests (cross-function escalation), the Operator Review Queue, Quick Send, async export, scheduled maintenance jobs, and semantic Answer Memory search. None of these get cut for good. The foundation is built to carry them, so they slot into the milestones after Phase 1 without a redesign.

SYSTEM DESIGN · CONTINUED

Built to Bend, Not Break

Two decisions matter most this early: the system should keep working when one part has a bad day, and adding the deferred features later should mean writing new code, not unpicking old code. Here's how the foundation handles both.

STAYS UP WHEN THINGS FAIL
  • AI runs as a background job with a timeout, a few retries, and a circuit breaker. If it stalls or goes down, the request falls back to manual routing instead of hanging.
  • A transactional outbox writes each state change and the notifications it triggers in the same database transaction, so nothing is lost if a worker dies mid-send.
  • The job queue retries with backoff and parks repeat failures in a dead-letter queue, so a bad job is visible and replayable, never silently dropped.
  • Idempotency keys and server-side status guards stop retries and double-clicks from creating duplicates or invalid states.
GROWS WITHOUT A REWRITE
  • The whole workflow is data-driven (Function Blocks, Process Lines, Steps), so new step types and routing rules are config and a handler, not a schema rewrite.
  • The AI sits behind one provider-agnostic interface, so swapping a model or adding a new one is a config change, not a refactor.
  • The event log and versioned prompts give clean seams for the analytics, Operator Review Queue, and semantic Answer Memory search that come later.
  • Deferred features like Support Requests, exports, and scheduled jobs attach as new NestJS modules on the same foundation.
